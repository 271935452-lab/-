【二期】PRD · 月底归档与运单底表

月底归档与运单底表 PRD

二期以下带角标的小节为与问题清单 14～22 及《域名说明》第 23、24 条等对齐的补充说明。

背景与目标

按固定节奏生成可追溯的月度批次，沉淀“当前批次运单明细+归档批次”，保障发放前后数据一致。

用户与场景

  • 财务：查看当前批次明细并导出。
  • 测试：验证调度、批次门槛、冻结逻辑。

二期字段定义与口径

  • 运单基础信息（蓝）：运单号、到货月份、收货计费重、件数、提单号、柜号、客户类型、业务员、客服、应收、业务员成本、毛利、销售产品、产品组、所属分公司；以上字段直接从运单管理取数。
  • 内陆费（运单汇总）：取值唯一来源为《数据字典》页签「需扣内陆费的服务商」——运单「服务商」与字典运单匹配值等值匹配；扣内陆费=是 → 金额 = 收货计费重 × 字典单价系数（默认 0.6）；宽表须具备收货计费重物理列或可映射为收货计费重的等价列（《07 产品回复》P16）。为否或未命中 → 本列 0。可与基数公式 #{SYS_INLAND_FEE} 对齐；支持导出单号明细。
  • 操作费（运单汇总）：按《基数说明》——件数 × 1.5，按运单计算后汇总展示；与「业务员成本」区分，参与毛利/净利润链路以实现为准。
  • 服务商：来自运单/承运回填，与服务商主数据对齐；筛选区支持模糊查询。
  • 回款情况（橙）：实收、回款率、回款截止核算时间（默认取账期自然月最后一天）。
  • 提成相关字段（紫）：表外毛利扣成本汇总、表外提成扣汇总、计算净利润、规则编码（第一版留空）。
  • 逻辑判断列（青）：是否整柜与客服是否有业务员并列于规则编码之后；表头统一为青色系（th-logic），数据单元格统一使用 td-logic，两列配色与字重一致，枚举为纯文本展示（不使用单独彩色标签）。是否整柜：按提单（HBL/BL）判断——若该提单在系统中仅关联本条运单（提单与运单 1:1），则展示「整柜」；若同一提单下存在多条运单则展示「非整柜」；空运等非海运场景是否展示「—」以实现字典为准。客服是否有业务员：业务员与客服为同一人 → 「无业务员」；非同一人 → 「有业务员」。
  • 批次追溯（灰）：归档批次号。
  • 开发口径 · 公式（先业务含义，后实现写法）
  • 毛利：用运单上的应收减去业务员成本，得到该运单毛利；币种、小数位与取整规则以财务主数据为准。
    gross_profit = receivable - sales_cost
  • 实收：汇总挂在本运单下、且账单状态为「已核销」的各笔账单金额之和。
    received_amount = SUM(bill.amount WHERE bill.waybill_id = 当前运单 AND bill.status = '已核销')
  • 回款率：实收除以应收；应收为 0 时不做除法，按产品约定记为空、0% 或「—」；展示层再格式化为百分比。
    collection_rate = IF(receivable = 0, NULL_OR_ZERO, received_amount / receivable)
  • 回款截止核算时间：默认取当前账期所在自然月的最后一日 23:59:59，作为回款统计的截止时点；若产品有单独约定则按约定覆盖。
    collection_cutoff_at = LAST_DAY_OF_MONTH(账期月份) 23:59:59
  • 表外毛利扣成本汇总：从「表外成本」导入数据中，筛出与本运单关联、且费用性质为「毛利扣」的行，对其金额求和（含正负）。
    offbook_gp_deduct_sum = SUM(import_line.amount WHERE import_line.waybill_id = 当前运单 AND import_line.fee_nature = '毛利扣')
  • 表外提成扣汇总：同上，费用性质为「提成扣」的导入行金额求和。
    offbook_comm_deduct_sum = SUM(import_line.amount WHERE import_line.waybill_id = 当前运单 AND import_line.fee_nature = '提成扣')
  • 计算净利润：毛利减去表外毛利扣成本汇总（即业务所称「毛利扣」汇总），得到用于提成链路的净利润口径。
    calc_net_profit = gross_profit - offbook_gp_deduct_sum
  • 当前批次：展示本轮计算载入的运单明细。
  • 归档批次：仅展示已冻结且可追溯批次主数据。
  • 数据范围（Q15）：统一为「相对数据账期所在月，往前连续 8 个自然月的运单集合」（与页顶调度说明一致）；示例行须同一算法生成。

交互说明

  • 页签：当前批次 / 归档批次。
  • 当前批次：运单宽表，支持刷新数据、导出。
  • 归档批次：支持导出批次明细。

与其他模块联动逻辑

  • 读取运单管理、业务数据统计、人员映射、月度调整、规则包命中结果生成当前批次；内陆费按《数据字典》原型页（数据字典-内陆费服务商与费用类型-MVP.html）服务商表匹配取值。
  • 当提成试算状态=已发放时，批次归档；否则不生成下一批次明细。
  • 归档后供报表/审计查询，原则上不回写历史结果。
  • 二期三地补发与主管门槛（域名说明第 23 条）：运单底表/宽表数据修正可能触发业务员重算与主管 T→T′ 分支；若已归档冻结，与「是否允许冲减主管、是否基于新快照另起调整单」策略一致，须与试算/提成调整模块联动，避免静默覆盖已冻结批次。

二期试算命中与账期（业务原则 16～17、22、域名 24）

  • 账期独立生效：人员映射、规则包、规则均以账期/生效区间独立版本参与命中；试算时须先命中人员映射 → 再命中规则包 → 最后命中规则，三者对同一账期同时有效才进入计提。
  • 列表状态：主档「已失效」不单独阻断数据账期仍落在生效区间内的补算（域名第 24 条）；归档宽表以批次账期与当时快照为准。
  • 规则包唯一性：同一提成角色 + 分公司 + 部门 + 小组在同一时间片内不得存在多个「生效中」规则包；冲突时阻断保存或清空关联，由人工处理。
  • 复用边界：基数方案可复用；规则包与规则不可复用（编码唯一，复制须生成新编码）。

二期规则包历史版本查看

  • 归档批次应落规则包版本ID / 规则版本集合摘要字段（若引擎已提供）；人工核对时打开规则包列表 → 历史版本与归档说明对照。

二期字段有/无时的匹配（本页 · 运单宽表 / 批次）

  • 运单行客服/业务员为空：逻辑列「客服是否有业务员」按 PRD 枚举；无法判断时以实现字典为准（默认不命中需双账号的规则）。
  • 销售产品为空：不参与依赖产品的规则命中展示；宽表仍保留行，规则编码可为空或标记未命中。
  • 服务商为空：无法匹配字典 → 内陆费列 0（或展示「—」以实现为准）。
  • 筛选区某条件为空：表示该维度不限。

二期字段取值来源（本页）

  • 运单基础列：运单管理 / OMS 当期镜像。
  • 内陆费、服务商、收货计费重、字典单价系数：《数据字典》（数据字典-内陆费服务商与费用类型-MVP.html）+ 运单扩展；操作费件数×1.5：运单扩展；与《基数说明》一致。
  • 表外毛利扣/提成扣汇总：月度调整导入按运单聚合。
  • 回款率、实收：账单中心核销结果。
  • 归档批次号、当前批次：归档调度任务写入。

同步《提成管理系统-问题清单-产品回复开发.docx》；与业务签字版冲突时以签字版为准。

  • Q13：调度触发统一为每月 24 日 00:00（与历史 PRD「23 日」不一致处以此为准）。
  • Q14～Q16：增量归档、数据范围示例、运单规则编码留空与回写等细则见上节「本次范围与后续规划」。

非功能性要求

  • 调度任务稳定重试、告警可观测。
  • 导出大数据量需异步任务与状态回传。
  • 批次冻结后具备不可篡改审计能力。

本次范围与后续规划

  • 本次（Q13）：每月 24 日 00:00 调度触发，跑相对数据账期往前连续 8 个自然月运单集合并生成/追加归档批次（与页顶调度条一致）。
  • Q14（建议）：发放可分批；归档侧可按日跑增量已发放数据入批次，不设单次截止直至跑完；状态机与表结构由开发另档细化。
  • Q15：各批次「数据范围」示例须同一窗口算法生成，避免 ARCH 行间口径矛盾。
  • Q16：运单宽表「规则编码」初始化可空，试算由引擎命中；归档/发放确认后可回写（防重复覆盖）；二期主数据收敛后可非空。
  • 后续：异常批次自动补跑、跨批次对比与差异分析。
