# 人员映射表列表页 PRD

## 1. 文档信息
- 页面名称：人员映射表列表页
- 文件原型：`personnel-mapping-list.html`
- 目标用户：产品运营、财务、薪酬结算、组织管理员
- 文档用途：供前端、后端、测试、产品评审使用
- 当前版本：V1.0
- 范围说明：本 PRD 仅覆盖“人员映射表列表页”，不包含新增/编辑详情页的完整表单设计

---

## 2. 页面目标

人员映射表列表页用于维护“系统账号”与“提成角色/组织/生效时间”的映射关系，支撑提成规则引擎在月度计算时准确识别：

1. 某个系统账号对应哪位员工
2. 员工当前属于哪个部门/小组
3. 员工在某段时间内具备哪些提成岗位
4. 员工当前在职、离职、即将失效等状态
5. 是否存在需要停用、续期、复制新映射等处理动作

该页面是规则命中前置数据页面，优先级高于规则配置页。

---

## 3. 业务背景

提成系统计算链路中，“人员映射”决定了“谁参与提成、以什么身份参与提成、在哪个时间段参与提成”。  
若该层数据不准确，会直接导致以下问题：

1. 规则命中到错误人员
2. 同一人出现重复提成身份
3. 已离职人员仍参与计提
4. 生效期断档或重叠导致试算异常
5. 无法追溯历史期间某员工对应的岗位身份

因此页面需同时兼顾：

1. 查询效率
2. 批量维护效率
3. 时间维度可追溯
4. 风险状态可识别

---

## 4. 页面定位

### 4.1 页面类型
- 后台管理列表页

### 4.2 页面定位
- 人员映射主入口
- 支持筛选、批量操作、状态识别、导入导出

### 4.3 不在本页面处理的能力
- 完整编辑表单配置
- 角色冲突详细修复
- 映射变更审批
- 映射历史 diff 详情

上述能力后续由“新增/编辑映射页”、“历史记录页”、“冲突分析页”承接。

## 7. 查询筛选区

### 7.1 查询条件

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| 系统账户名称 | 输入框 | 否 | 支持模糊查询 |
| 姓名 | 输入框 | 否 | 支持模糊查询 |
| 部门 | 输入框 | 否 | 支持部门/小组模糊查询 |
| 生效开始时间 | 日期选择器 | 否 | 查询映射生效起始时间 >= 指定时间 |
| 生效结束时间 | 日期选择器 | 否 | 查询映射生效起始时间 <= 指定时间 |

### 7.2 查询规则

#### 系统账户名称
- 匹配方式：`like`
- 忽略大小写（由后端统一处理）

#### 姓名
- 匹配方式：`like`

#### 部门
- 匹配方式：`like`
- 支持按部门名或小组名检索

#### 生效时间范围
- 查询逻辑：
  - 若只传生效开始时间：查询 `effective_start >= start`
  - 若只传生效结束时间：查询 `effective_start <= end`
  - 若同时传入：查询 `effective_start between start and end`

> 注：当前原型字段命名为“生效开始时间范围”，但列表实际字段为“生效起始时间”。后端实现以映射记录的 `effective_start` 为准。

### 7.3 功能按钮

#### 重置
- 清空所有查询条件
- 回到默认第一页
- 重新拉取默认列表

#### 查询
- 使用当前条件发起查询
- 默认回到第 1 页

## 9. 列表区设计

### 9.1 表格字段

| 列名 | 字段key | 类型 | 说明 |
|---|---|---|---|
| 勾选框 | checked | checkbox | 支持单选/全选 |
| 系统账户名称 | account_name | 文本 | 主展示字段 |
| 员工姓名 | employee_name | 文本 | 员工真实姓名 |
| 所属部门/小组 | dept_name | 文本 | 组织信息 |
| 生效起始时间 | effective_start | 日期 | 当前映射生效时间 |
| 失效截止时间 | effective_end | 日期 | 当前映射失效时间 |
| 入职时间 | hire_date | 日期 | 来源于员工信息 |
| 离职时间 | leave_date | 日期 / `-` | 若未离职显示 `-` |
| 人员状态 | employee_status | 标签 | 在职 / 即将失效 / 离职 |
| 提成岗位 | commission_roles | 标签组 | 支持多角色展示 |
| 操作 | actions | 链接按钮 | 查看/编辑/停用等 |

### 9.2 字段展示说明

#### 系统账户名称
- 主信息：系统账户名称
- 副信息：`账号ID：xxxxx`

#### 人员状态
状态展示建议：

| 状态 | 标签样式 | 规则 |
|---|---|---|
| 在职 | 绿色 | 当前日期 < 离职日期，且映射未失效 |
| 即将失效 | 黄色 | `effective_end` 距当前日期小于等于预警阈值（建议 30 天） |
| 离职 | 灰色 | 存在离职日期且已离职 |
| 停用 | 红色 | 映射记录被手工停用 |

> 当前原型中有 `tag-danger` 颜色定义，正式开发建议补齐“停用”状态字段，以支持红色标签。

#### 提成岗位
- 支持多标签展示
- 示例：
  - `直客客服`
  - `有业务员客服`
  - `同行业务员`

> 当前产品讨论里“有业务员客服 / 无业务员客服”后续建议逐步沉淀为场景字段 `service_mode`，但列表页在 MVP 中仍允许以岗位标签方式展示，避免用户认知断层。

## 13. 状态设计

建议后端为人员映射记录维护两个维度状态：

### 13.1 人员主状态 `employee_status`
- 在职
- 离职

### 13.2 映射状态 `mapping_status`
- 生效中
- 即将失效
- 已失效
- 已停用

前端列表页最终展示状态优先级建议：

1. 若 `mapping_status = 已停用`，展示停用
2. 否则若 `employee_status = 离职`，展示离职
3. 否则若 `mapping_status = 即将失效`，展示即将失效
4. 否则展示在职

## 15. 字段校验与业务规则

虽然本页面主要为列表页，但前后端仍需遵守以下业务校验逻辑：

### 15.1 唯一性建议
同一员工、同一系统账户、同一提成岗位，在同一有效期内不允许出现完全重叠映射。

### 15.2 时间有效性
- `effective_start <= effective_end`
- `hire_date <= effective_start` 建议校验
- 若有 `leave_date`，则 `effective_start <= leave_date`

### 15.3 离职处理
- 已离职人员允许保留历史映射
- 默认不可新增新的长期有效映射，除非业务允许返聘场景

### 15.4 岗位多值
- 一个员工可存在多个提成岗位
- 若存在互斥岗位，冲突检查在新增/编辑页完成

# 人员映射表列表页 PRD

## 1. 文档信息

- 产品名称：人员映射表
- 页面类型：后台管理列表页
- 适用对象：产品、前端、后端、测试
- 关联原型：`personnel-mapping-list.html`
- 文档目的：明确人员映射表列表页的业务目标、页面结构、字段定义、交互规则、接口诉求与验收标准，便于开发联调和测试验收

## 2. 背景与目标

当前业务需要维护系统账户与员工、组织、岗位、规则包之间的映射关系，用于后续角色识别、规则归属、流程分发及统计分析。

现阶段问题主要包括：

- 系统账户与员工岗位关系分散，缺少统一维护入口
- 生效时间和失效时间缺少集中管理，容易出现时间重叠或失效遗漏
- 离职人员历史映射与当前有效映射混杂，查询成本高
- 业务希望重点关注“人员节点是否配置正确”，其余校验和处理由系统兜底

本页面目标：

- 提供统一的人员映射查询与维护入口
- 支持按系统账户名称、姓名、部门、生效时间范围快速检索
- 支持新增、编辑、失效、批量导入、导出等操作
- 明确当前是否生效、人员状态、规则归属等关键信息
- 为后续角色配置、工单流转、提成规则匹配提供基础数据支撑

## 3. 页面范围

本期范围包含：

- 搜索区
- 顶部操作区
- 统计卡片
- 列表表格
- 分页
- 行内操作入口
- 页面级静态提示信息

本期不包含：

- 新增/编辑弹窗的详细表单实现
- 批量导入模板下载页

- 已失效数据：显示“查看、编辑、启用”
- 异常数据：显示“查看、补齐规则、失效”
- 若用户无编辑权限，仅显示“查看”

## 11. 核心业务规则

### 11.1 生效规则

- 当前时间大于等于生效开始时间，且小于等于生效截止时间时，状态为“生效中”
- 当前时间小于生效开始时间时，状态为“未生效”
- 当前时间大于生效截止时间时，状态为“已失效”
- 若生效截止时间为空，则视为长期有效

### 11.2 即将到期规则

- 当前状态为生效中
- 且生效截止时间存在
- 且距离截止时间小于等于 30 天

### 11.3 批量失效规则

- 仅允许对选中的有效数据执行批量失效
- 批量失效需二次确认
- 失效后应记录操作人和操作时间

### 11.4 导入校验规则

- 系统账户名称不能为空
- 员工姓名不能为空
- 所属部门不能为空
- 人员岗位不能为空
- 生效开始时间不能为空
- 生效开始时间不能晚于生效截止时间
- 同一系统账户在同一时间范围内不允许存在重复生效映射
- 离职人员默认不允许新增长期有效映射

### 11.5 异常判定规则

满足以下任一情况时可标记为异常：

- 规则包缺失
- 组织路径缺失
- 必填字段为空
- 时间区间与现有有效映射重叠
- 岗位与规则不匹配

## 12. 交互说明

### 12.1 查询交互

- 点击“查询”后，刷新统计卡片和列表数据
- 查询条件变化后，不自动触发查询
- 点击“重置”后恢复默认条件并重新加载默认列表

### 12.2 表格交互

- 支持多选
- 表头勾选框支持全选当前页数据
- 长文本字段允许换行展示
- 操作列固定在表格右侧可作为后续优化项

### 12.3 分页交互

- 支持分页展示
- 默认每页 10 条或 20 条，按系统通用规范处理
- 翻页后保留查询条件

### 12.4 空状态

当无查询结果时，页面显示空状态文案：

`暂无符合条件的人员映射数据`

### 12.5 异常提示

- 批量操作未勾选数据时，提示“请先选择数据”
- 时间范围非法时，提示“开始时间不能晚于结束时间”
- 导入失败时，返回失败原因明细

## 13. 接口需求建议

以下为推荐接口拆分，最终以后端实际设计为准。

### 13.1 列表查询接口

- 接口用途：查询人员映射列表
- 请求参数：
  - 系统账户名称
  - 姓名
  - 部门 ID
  - 生效开始时间
  - 生效结束时间
  - 人员状态
  - 页码
  - 每页条数
- 返回参数：
  - 列表数据
  - 总条数
  - 当前页
  - 每页条数
  - 统计卡片数据

### 13.2 新增映射接口

- 接口用途：新增一条人员映射
- 请求参数：
  - 系统账户名称
  - 员工 ID / 员工姓名
  - 部门 ID
  - 规则 ID
  - 岗位
  - 生效开始时间
  - 生效截止时间
  - 备注

### 13.3 编辑映射接口

- 接口用途：修改已有映射
- 需返回冲突校验结果

### 13.4 失效接口

- 接口用途：单条失效或批量失效
- 请求参数：
  - 映射 ID 列表
  - 失效原因

### 13.5 导出接口

- 接口用途：导出当前查询结果
- 导出内容与列表字段保持一致

### 13.6 导入接口

- 接口用途：批量导入映射关系
- 返回导入成功数量、失败数量、失败原因明细

## 16. 验收标准

### 16.1 功能验收

- 能按系统账户名称、姓名、部门、生效时间范围进行查询
- 查询结果与统计卡片口径一致
- 能执行新增、编辑、失效、批量导入、导出
- 行内操作展示符合数据状态规则
- 分页正常，翻页后条件不丢失

### 16.2 数据验收

- 生效中、未生效、已失效、即将到期状态计算正确
- 长期有效数据展示正确
- 离职人员数据状态展示正确
- 异常数据可被识别并展示异常状态

### 16.3 异常验收

- 非法时间范围有明确提示
- 批量失效未选择数据时有提示
- 导入失败时可返回失败原因
- 重复映射和时间重叠能被系统拦截
