# 人员映射弹窗 PRD

## 1. 文档信息
- 文档名称：人员映射新增/编辑弹窗 PRD
- 关联页面：`personnel-mapping-list.html`
- 适用范围：人员映射列表页中的“新增映射”和“编辑映射”弹窗
- 面向对象：前端开发、后端开发、测试、产品
- 当前版本：V1.0

---

## 2. 目标说明

该弹窗用于完成“人员映射”的新增与编辑操作，是人员映射列表页的核心维护入口。

主要解决以下业务问题：

1. 为系统账号新增提成角色映射
2. 对已有映射进行部门、角色、生效时间、状态等调整
3. 支持在同一弹窗中维护多角色映射
4. 支持客服类角色结合 `service_mode` 管理“有业务员客服 / 无业务员客服 / 全部场景”
5. 在保存前进行基础字段校验和业务规则提示

---

## 3. 功能范围

### 3.1 本期范围
- 新增映射弹窗
- 编辑映射弹窗
- 弹窗字段填写、回填、关闭、保存按钮交互
- 前端基础校验
- 保存草稿和确认保存按钮展示

### 3.2 不在本期范围
- 真正的接口联调提交
- 保存后的列表自动刷新逻辑细节
- 历史版本对比
- 冲突检测结果页
- 审批流

## 4. 弹窗入口

### 4.1 新增映射入口
- 页面顶部按钮：`+ 新增映射`
- 点击后打开弹窗
- 弹窗标题：`新增映射`
- 副标题：`新增系统账号与提成角色的映射关系`
- 主按钮文案：`确认新增`

### 4.2 编辑映射入口
- 列表行内按钮：`编辑`
- 点击后打开弹窗
- 弹窗标题：`编辑映射`
- 副标题：`编辑当前系统账号的提成角色、生效时间与人员状态`
- 主按钮文案：`确认保存`

---

## 5. 弹窗关闭方式

支持以下关闭方式：

1. 点击右上角关闭按钮
2. 点击底部 `取消`
3. 点击遮罩层关闭
4. 键盘 `Esc` 关闭

### 关闭规则
- 当前版本仅静态原型，不做“未保存提示”
- 正式开发建议：
  - 若表单有改动未保存，关闭时弹出确认提示

---

## 6. 弹窗结构

弹窗分为 4 个区域：

1. 头部区域
2. 基础信息区域
3. 提成角色区域
4. 时间信息区域
5. 备注与校验提示区域
6. 底部操作区

## 7. 字段设计

## 7.1 基础信息区域

### 字段 1：系统账户名称
- 字段 key：`accountName`
- 组件：输入框
- 是否必填：是
- 占位文案：`请输入系统账户名称`
- 新增默认值：空
- 编辑回填：从当前行 `data-account` 回填

#### 校验
- 不可为空
- 去除首尾空格后校验
- 建议限制最大长度 50
- 建议做唯一性校验（接口层完成）

---

### 字段 2：员工姓名
- 字段 key：`employeeName`
- 组件：输入框
- 是否必填：是
- 占位文案：`请输入员工姓名`
- 新增默认值：空
- 编辑回填：从当前行 `data-name` 回填

#### 校验
- 不可为空
- 最大长度建议 30

---

### 字段 3：所属部门/小组
- 字段 key：`deptName`
- 组件：输入框
- 是否必填：是
- 占位文案：`如：深圳客服部 / 一组`
- 新增默认值：空
- 编辑回填：从当前行 `data-dept` 回填

#### 校验
- 不可为空
- 最大长度建议 100

---

### 字段 4：人员状态
- 字段 key：`employeeStatus`
- 组件：下拉选择
- 是否必填：是
- 枚举值：
  - `在职`
  - `离职`
- 新增默认值：`在职`
- 编辑回填：从当前行 `data-status` 回填

#### 业务含义
- 表示员工当前主状态
- 不等同于映射状态

---

### 字段 5：service_mode
- 字段 key：`serviceMode`
- 组件：下拉选择
- 是否必填：是
- 枚举值：
  - `全部场景`
  - `有业务员客服`
  - `无业务员客服`
- 新增默认值：`全部场景`
- 编辑回填逻辑：
  - 若角色中包含 `有业务员客服`，默认回填 `有业务员客服`
  - 若角色中包含 `无业务员客服`，默认回填 `无业务员客服`
  - 否则回填 `全部场景`

#### 业务说明
- 用于区分客服相关场景
- 非客服角色默认可用 `全部场景`
- 后续可扩展更多分片场景

---

## 7.2 提成角色区域

### 字段：提成角色
- 字段 key：`roles`
- 组件：多选 checkbox
- 是否必填：是，至少选 1 项
- 角色枚举：
  - `直客业务员`
  - `直客主管`
  - `直客客服`
  - `客服中心主管`
  - `同行业务员`
  - `同行经理`
  - `渠道客服`

### 新增默认值
- 全部未选中

### 编辑回填逻辑
- 从当前行 `data-role` 读取
- 使用逗号分割后逐项勾选

### 业务规则
- 一个员工可同时拥有多个提成角色
- 但同一时间范围内不允许出现业务冲突角色组合
- 当前弹窗仅做提示，不在前端实现完整冲突计算

### 前端校验
- 至少勾选一个角色

---

## 7.3 时间信息区域

### 字段 1：生效起始时间
- 字段 key：`effectiveStart`
- 组件：日期选择器
- 是否必填：是
- 新增默认值：空
- 编辑回填：从 `data-start` 回填

### 字段 2：失效截止时间
- 字段 key：`effectiveEnd`
- 组件：日期选择器
- 是否必填：否
- 新增默认值：空
- 编辑回填逻辑：
  - 若 `data-end = 2099-12-31`，前端回填为空，表示长期有效
  - 否则回填真实日期

### 字段 3：入职时间
- 字段 key：`hireDate`
- 组件：日期选择器
- 是否必填：否
- 新增默认值：空
- 编辑回填：从 `data-hire` 回填

### 字段 4：离职时间
- 字段 key：`leaveDate`
- 组件：日期选择器
- 是否必填：否
- 新增默认值：空
- 编辑回填：从 `data-leave` 回填

### 时间校验规则

#### 校验 1
- 生效起始时间不能为空

#### 校验 2
- 若填写失效截止时间，则 `effectiveStart <= effectiveEnd`

#### 校验 3
- 若填写入职时间，则建议 `hireDate <= effectiveStart`

#### 校验 4
- 若填写离职时间，则建议 `effectiveStart <= leaveDate`

#### 校验 5
- 离职人员默认不允许新增长期有效映射

---

## 7.4 备注与校验提示区域

### 字段：备注说明
- 字段 key：`remark`
- 组件：多行输入框
- 是否必填：否
- 新增默认值：空
- 编辑默认值：`由列表页发起编辑，可补充本次变更说明。`

### 用途
- 记录变更原因
- 记录特殊业务说明
- 供审计和后续排障使用

### 字数限制
- 建议最大长度 500

## 9. 新增逻辑

## 9.1 打开逻辑
- 点击 `+ 新增映射`
- 执行 `openCreateModal()`

## 9.2 表单初始化逻辑
- 标题设置为 `新增映射`
- 副标题设置为 `新增系统账号与提成角色的映射关系`
- 主按钮设置为 `确认新增`
- 所有输入框清空
- 人员状态默认 `在职`
- `service_mode` 默认 `全部场景`
- 所有角色复选框取消选中

## 9.3 提交逻辑

前端需执行以下基础校验：

1. 系统账户名称不能为空
2. 员工姓名不能为空
3. 所属部门/小组不能为空
4. 至少选择一个提成角色
5. 生效起始时间不能为空
6. 若失效截止时间有值，则生效开始时间不能晚于截止时间
7. 若人员状态为离职且失效截止时间为空，给出提示

校验通过后提交新增接口。

## 9.4 新增成功后
- 关闭弹窗
- 列表刷新
- toast 提示：`新增成功`

---

## 10. 编辑逻辑

## 10.1 打开逻辑
- 点击行内 `编辑`
- 执行 `openEditModal(trigger)`

## 10.2 回填逻辑

### 字段回填来源
- `data-account` -> 系统账户名称
- `data-name` -> 员工姓名
- `data-dept` -> 所属部门/小组
- `data-status` -> 人员状态
- `data-role` -> 角色字符串
- `data-start` -> 生效起始时间
- `data-end` -> 失效截止时间
- `data-hire` -> 入职时间
- `data-leave` -> 离职时间

### 角色回填
- 将 `data-role` 用逗号分割
- 循环勾选匹配的 checkbox

### service_mode 回填
- 若角色文案中包含 `有业务员客服`，则选中 `有业务员客服`
- 若角色文案中包含 `无业务员客服`，则选中 `无业务员客服`
- 否则选中 `全部场景`

### 长期有效逻辑
- 若后端返回的失效时间为 `2099-12-31`
- 前端表单回填为空值，表示长期有效

## 10.3 编辑提交逻辑
- 沿用新增弹窗的校验规则
- 编辑提交时保留原记录 ID（实际接口提交需要）
- 保存成功后关闭弹窗并刷新列表

## 10.4 编辑成功后
- toast 提示：`保存成功`


## 12. 接口建议

## 12.1 新增接口

### 请求
`POST /api/commission/personnel-mappings`

### 请求体示例
```json
{
  "accountName": "black-cat-01",
  "employeeName": "黄丽青",
  "deptName": "深圳客服部",
  "employeeStatus": "ACTIVE",
  "serviceMode": "WITH_SALES",
  "roles": ["直客客服"],
  "effectiveStart": "2026-01-01",
  "effectiveEnd": null,
  "hireDate": "2023-03-15",
  "leaveDate": null,
  "remark": "新增直客客服映射"
}
```

## 12.2 编辑接口

### 请求
`PUT /api/commission/personnel-mappings/{id}`

### 请求体示例
```json
{
  "deptName": "深圳客服部",
  "employeeStatus": "ACTIVE",
  "serviceMode": "NO_SALES",
  "roles": ["直客客服"],
  "effectiveStart": "2026-01-01",
  "effectiveEnd": "2026-12-31",
  "hireDate": "2023-03-15",
  "leaveDate": null,
  "remark": "调整为无业务员客服场景"
}
```

## 12.3 后端校验建议

1. 系统账户名称唯一性校验
2. 同账户同角色在时间范围内不可重叠
3. 离职人员不可新增长期有效映射
4. 客服类角色若配置了 `service_mode`，需校验取值合法

## 13. 前端校验提示文案建议

| 场景 | 提示文案 |
|---|---|
| 系统账户名称为空 | 请输入系统账户名称 |
| 员工姓名为空 | 请输入员工姓名 |
| 所属部门为空 | 请输入所属部门/小组 |
| 未选择角色 | 请至少选择一个提成角色 |
| 生效开始时间为空 | 请选择生效起始时间 |
| 生效开始晚于截止时间 | 生效起始时间不能晚于失效截止时间 |
| 离职状态但未填离职时间 | 当前人员状态为离职，请补充离职时间 |

## 14. 交互说明

### 14.1 新增与编辑区别

| 维度 | 新增 | 编辑 |
|---|---|---|
| 标题 | 新增映射 | 编辑映射 |
| 副标题 | 新增系统账号与提成角色的映射关系 | 编辑当前系统账号的提成角色、生效时间与人员状态 |
| 主按钮文案 | 确认新增 | 确认保存 |
| 表单默认值 | 空值 + 默认状态 | 回填当前记录 |

### 14.2 遮罩关闭
- 点击弹窗外遮罩关闭

### 14.3 Esc关闭
- 按 Esc 关闭

### 14.4 滚动行为
- 弹窗内容较长时内部滚动，不影响页面整体滚动
